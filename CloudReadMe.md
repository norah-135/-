# Cloud Backend



##  المعمارية العامة  (Architecture Overview)

تم تقسيم السحابة إلى خدمتين مستقلتين تماماً لضمان عدم تأثر نظام الإنقاذ الحرج بأي عمليات تشغيلية أخرى:

1. **باك اند التطبيق (App Backend):** يدار عبر بروتوكول HTTP التقليدي ومسؤول عن العمليات غير الحرجة زمنيّاً (تسجيل الحسابات، تعديل الملف الطبي، الحفظ والتعديل).
2. **محرك الطوارئ والـ IoT (IoT/Emergency Core):** يدار عبر بروتوكولات الـ IoT وبنية أحداث حية (Event-Driven)، وهو المسؤول عن استقبال الاستغاثات من المسارات الأربعة، وإدارة جلسة الطوارئ الحية، وتوجيه طائرات الدرون.

---

##  الخدمات المستخدمة في AWS (AWS Services Stack)

* **AWS IoT Core:** وسيط الرسائل (Message Broker) الأساسي الذي يستقبل تدفق بيانات الـ MQTT من التطبيقات وبوابات الـ IoT في محطات LoRa.
* **AWS API Gateway:** نقطة الدخول الآمنة لاستقبال طلبات الـ REST APIs القادمة من تطبيق الجوال (القسم غير الحرج) أو لوحة تحكم المستشفى.
* **AWS Lambda:** الدوال البرمجية (Serverless Functions) التي تحتوي على المنطق الخبير للـ Fail-Safe وتحديد أقرب درون وتوجيه الأوامر.
* **Amazon DynamoDB:** قاعدة بيانات NoSQL فائقة السرعة ومنخفضة التأخير لتخزين ملفات المستخدمين، سجلات البلاغات، وحالات أسطول الطائرات.
* **AWS IAM:** لإدارة الصلاحيات والتوثيق الداخلي المشفر بين الخدمات دون الحاجة لتمرير البيانات عبر الإنترنت العام.

---

##  شجرة قنوات الـ MQTT (MQTT Topics Hierarchy)

تعتمد المنظومة على القنوات التالية لتوجيه البيانات لحظياً بين أطراف النظام:

| القناة (Topic) | الناشر (Publisher) | المشترك (Subscriber) | الوظيفة |
| :--- | :--- | :--- | :--- |
| `aeromedic/patients/{id}/alerts` | التطبيق / محطة LoRa | AWS IoT Core | إرسال إشارة استغاثة طوارئ حرجة |
| `aeromedic/drones/{id}/telemetry` | طائرة الدرون (Pi) | سحابة الطوارئ | تحديث موقع الطائرة، الارتفاع، والبطارية |
| `aeromedic/patients/{id}/drone_status` | سحابة الطوارئ | تطبيق الجوال | استقبال المريض لموقع الطائرة الحي على الخريطة |

## 📊 مخطط قاعدة البيانات (Database Schema - DynamoDB)

تتكون ذاكرة النظام من ثلاثة جداول أساسية مصممة للربط اللحظي عند الأزمات:

### 1. جدول المستخدمين (Users Table)
* `patient_id` (مفتاح أساسي - String)
* `name` (String)
* `medical_history`: (Map يحتوي على الحساسية الحرجة، الربو، فصيلة الدم)
* `created_at` (Timestamp)

### 2. جدول الطائرات والمنصات (Drones Table)
* `drone_id` (مفتاح أساسي - String)
* `station_location`: (Map يحتوي على خطوط الطول والعرض لمنصة الإطلاق)
* `status` (String: Available | In-Flight | Maintenance)
* `battery_level` (Number)

### 3. جدول جلسات الطوارئ النشطة (Active Sessions Table)
* `session_id` (مفتاح أساسي - String)
* `patient_id` (String)
* `drone_id` (String - المحدد للمهمة)
* `current_location`: (Map لإحداثيات الحادث الحالية)
* `biometrics`: (آخر قراءة SpO2 )
* `session_status` (String: Triggered | En-Route | Arrived | Resolved)

##  دورة حياة البيانات وتدفق الـ Fail-Safe (Data Pipeline)

عند رصد حالة طوارئ وانتهاء الـ 20 ثانية التحذيرية في التطبيق، يتم تنفيذ التدفق التالي داخل السحابة:

1. **الاستقبال:** تصل حزمة البيانات (Payload) إلى `AWS IoT Core` عبر القناة المخصصة.
2. **الاستدعاء اللحظي (On-Demand Fetching):** تقوم دالة الطوارئ (`Emergency Lambda`) بالعمل، وتطلب فوراً عبر صلاحيات `AWS IAM` الملف الطبي للمريض من جدول المستخدمين التابع لباك اند التطبيق.
3. **التحقق الآمن (Fail-Safe Check):** يطابق المنطق الخبير نوع الطوارئ مع ملف المريض؛ إذا تأكد الأمان، ينتقل للخطوة التالية.
4. **التوجيه الذكي:** تحسب الدالة المسافة بين موقع الحادث وجميع الطائرات المتوفرة (Available) في جدول الطائرات، وتختار الأقرب.
5. **الأمر:** ترسل السحابة أمر الإقلاع الآلي (`Takeoff Order`) لمحطة الدرون المعنية عبر الـ API الخاص بـ `PX4 Autopilot`.
